JVM Debug MCP

An MCP (Model Context Protocol) server that enables AI assistants to debug Java and Kotlin applications through the JDWP (Java Debug Wire Protocol).

Breakpoint Debugging

Set breakpoints by class name and line number, step through code, and inspect variables.

Multi-Thread Support

Debug multiple threads independently with per-thread suspend, resume, and step operations.

Build System Integration

Auto-detect and launch Gradle and Maven projects with debug flags automatically configured.

22 MCP Tools

Full debug workflow from project detection to variable inspection, all accessible via MCP.

Installation

Prerequisites

  • Node.js 18+
  • Java JDK 11+ (for the target application)
  • Gradle or Maven (optional, for build system integration)

Install

git clone https://github.com/Riku-KANO/jvm-debug-mcp.git
cd jvm-debug-mcp
pnpm install
pnpm run build

MCP Client Configuration

Add the following to your MCP client configuration (e.g., Claude Desktop claude_desktop_config.json):

{
  "mcpServers": {
    "jvm-debug": {
      "command": "node",
      "args": ["/path/to/jvm-debug-mcp/dist/index.js"]
    }
  }
}

Quick Start

Option A: Launch & Debug a Project

For Gradle or Maven projects, the MCP server can build, launch, and attach the debugger automatically.

  1. Detect the project 
    detect_project({ projectDir: "/path/to/your/project" })
  2. Launch with debug agent 
    launch({ projectDir: "/path/to/your/project" })

    This builds the project and starts it with JDWP enabled on port 5005. The launch tool verifies that the JDWP port is actually accepting connections before reporting success (up to 10 seconds).

  3. Connect the debugger 
    connect({ host: "localhost", port: 5005 })
  4. Set a breakpoint and debug 
    set_breakpoint({ className: "com.example.MyController", line: 25 })
// Trigger the code path, then:
get_stack_trace({})
get_variables({})

Option B: Attach to an Existing JVM

Start your application with the JDWP agent, then connect:

# Start your application with JDWP
java -agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005 -jar app.jar

# Then use the connect tool
connect({ host: "localhost", port: 5005 })

Build System Support

Build System Detection Default Task Debug Args Injection
Gradle (Groovy DSL) build.gradle run / bootRun Init script (--init-script)
Gradle (Kotlin DSL) build.gradle.kts run / bootRun Init script (--init-script)
Maven pom.xml exec:java (default) / spring-boot:run MAVEN_OPTS / -Dspring-boot.run.jvmArguments

Spring Boot

Spring Boot projects are automatically detected. The server checks for Spring Boot plugins in your build files and selects the appropriate run task:

  • Gradle: Uses bootRun when org.springframework.boot plugin is detected
  • Maven: Uses spring-boot:run when spring-boot-maven-plugin is detected, exec:java when exec-maven-plugin is detected, and defaults to exec:java otherwise
Gradle Init Script: JDWP arguments are injected only into JavaExec tasks (like bootRun) via a temporary Gradle init script. This ensures the Gradle daemon JVM is not affected.

Breakpoints

Set breakpoints using fully qualified class names and line numbers:

set_breakpoint({
  className: "com.example.UserService",
  line: 42,
  suspendPolicy: "thread"  // "thread" (default) or "all"
})

Suspend Policies

Policy Behavior Use Case
thread Only the thread that hit the breakpoint is suspended Multi-thread debugging, API servers
all All threads are suspended Deterministic debugging, race condition investigation

Multi-Thread Debugging

The server supports true multi-thread debugging. When using suspendPolicy: "thread", each thread can be independently controlled:

  1. Set breakpoints on different handlers 
    set_breakpoint({ className: "com.example.Api1Controller", line: 30 })
set_breakpoint({ className: "com.example.Api2Controller", line: 25 })
  2. Trigger both endpoints (e.g., via curl)

    Both threads hit their breakpoints independently.

  3. Inspect each thread 
    get_threads({})  // See all threads and their status
get_stack_trace({ threadId: "42" })
get_variables({ threadId: "42" })
get_stack_trace({ threadId: "43" })
get_variables({ threadId: "43" })
  4. Step through each thread independently 
    step_over({ threadId: "42" })  // Thread 43 stays paused
step_over({ threadId: "43" })  // Thread 42 stays paused

Variable Inspection

Inspect local variables, object fields, and array elements:

// Get local variables in the current frame
get_variables({ threadId: "42", frameIndex: 0 })
// Output:
//   String name = "Alice"
//   int count = 5
//   UserService userService = Object@1234

// Drill into an object
inspect_object({ objectId: "1234" })
// Output:
//   String name = "UserService"
//   Map<String, User> cache = Object@5678

// Inspect array elements
inspect_array({ arrayId: "9012", startIndex: 0, length: 10 })

Protocol Stack

This server implements the JDWP (Java Debug Wire Protocol) directly over TCP, without relying on JDI (Java Debug Interface). This allows it to run in a pure Node.js environment.

LayerTechnologyRole
Client InterfaceMCP (stdio)Communication with AI assistants
Debug ProtocolJDWP (TCP)Binary protocol for JVM debugging
TargetJVM (JVMTI agent)Java/Kotlin application being debugged

Troubleshooting

Connection refused

Ensure the target JVM is started with the JDWP agent and the port matches:

-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005

Class not found when setting breakpoint

  • Use the fully qualified class name (e.g., com.example.MyClass, not MyClass)
  • The class must be loaded by the JVM. If using deferred breakpoints, trigger class loading first.

Port already in use

Another process may be using port 5005. Either stop it or specify a different port:

launch({ projectDir: "...", port: 5006 })

Launch succeeds but connect fails

If launch reports failure with "JDWP port is not listening", it means the process started but did not expose a JDWP debug port. This typically happens when:

  • The build system task does not pass JDWP arguments to the target JVM (e.g., running a plain Maven project without exec-maven-plugin)
  • The application takes longer than 10 seconds to start. In this case, try connecting manually with connect after waiting.

Gradle daemon conflict

This server uses a Gradle init script to inject debug flags only into JavaExec tasks. The Gradle daemon itself is never affected. If you still encounter issues, try ./gradlew --stop before launching.
