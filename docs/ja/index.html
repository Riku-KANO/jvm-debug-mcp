JVM Debug MCP

JDWP (Java Debug Wire Protocol) を通じて、AI アシスタントが Java/Kotlin アプリケーションをデバッグできるようにする MCP サーバーです。

ブレークポイントデバッグ

クラス名と行番号でブレークポイントを設定し、コードをステップ実行して変数を確認できます。

マルチスレッド対応

スレッドごとに独立してサスペンド・レジューム・ステップ実行が可能です。

ビルドシステム統合

Gradle / Maven プロジェクトを自動検出し、デバッグフラグ付きで起動します。

22 の MCP ツール

プロジェクト検出から変数確認まで、デバッグワークフロー全体を MCP 経由で操作できます。

インストール

前提条件

  • Node.js 18 以上
  • Java JDK 11 以上（デバッグ対象アプリケーション用）
  • Gradle または Maven（ビルドシステム統合を利用する場合）

セットアップ

git clone https://github.com/Riku-KANO/jvm-debug-mcp.git
cd jvm-debug-mcp
npm install
npm run build

MCP クライアント設定

MCP クライアントの設定ファイル（例: Claude Desktop の claude_desktop_config.json）に以下を追加します：

{
  "mcpServers": {
    "jvm-debug": {
      "command": "node",
      "args": ["/path/to/jvm-debug-mcp/dist/index.js"]
    }
  }
}

クイックスタート

方法 A: プロジェクトの起動とデバッグ

Gradle / Maven プロジェクトの場合、MCP サーバーがビルド・起動・デバッガ接続を自動で行います。

  1. プロジェクトを検出 
    detect_project({ projectDir: "/path/to/your/project" })
  2. デバッグエージェント付きで起動 
    launch({ projectDir: "/path/to/your/project" })

    プロジェクトをビルドし、JDWP をポート 5005 で有効にして起動します。launch ツールは JDWP ポートが実際に接続可能な状態になったことを確認してから成功を返します（最大 10 秒）。

  3. デバッガを接続 
    connect({ host: "localhost", port: 5005 })
  4. ブレークポイントを設定してデバッグ 
    set_breakpoint({ className: "com.example.MyController", line: 25 })
// 該当コードパスを実行した後:
get_stack_trace({})
get_variables({})

方法 B: 起動済みの JVM に接続

アプリケーションを JDWP エージェント付きで起動してから接続します：

# JDWP 付きでアプリケーションを起動
java -agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005 -jar app.jar

# connect ツールで接続
connect({ host: "localhost", port: 5005 })

ビルドシステム対応

ビルドシステム 検出ファイル デフォルトタスク デバッグ引数の注入方法
Gradle (Groovy DSL) build.gradle run / bootRun Init script (--init-script)
Gradle (Kotlin DSL) build.gradle.kts run / bootRun Init script (--init-script)
Maven pom.xml exec:java（デフォルト） / spring-boot:run MAVEN_OPTS / -Dspring-boot.run.jvmArguments

Spring Boot

Spring Boot プロジェクトは自動的に検出されます。ビルドファイル内の Spring Boot プラグインを確認し、適切な実行タスクを選択します：

  • Gradle: org.springframework.boot プラグインを検出すると bootRun を使用
  • Maven: spring-boot-maven-plugin を検出すると spring-boot:run、exec-maven-plugin を検出すると exec:java、いずれも検出されない場合はデフォルトで exec:java を使用
Gradle Init Script: JDWP 引数は一時的な Gradle init script を通じて JavaExec タスク（bootRun など）にのみ注入されます。Gradle デーモンの JVM には影響しません。

ブレークポイント

完全修飾クラス名と行番号を指定してブレークポイントを設定します：

set_breakpoint({
  className: "com.example.UserService",
  line: 42,
  suspendPolicy: "thread"  // "thread"（デフォルト）または "all"
})

サスペンドポリシー

ポリシー 動作 ユースケース
thread ブレークポイントに到達したスレッドのみ停止 マルチスレッドデバッグ、API サーバー
all 全スレッドが停止 確定的なデバッグ、競合状態の調査

マルチスレッドデバッグ

本サーバーは真のマルチスレッドデバッグに対応しています。 suspendPolicy: "thread" を使用すると、各スレッドを独立して制御できます：

  1. 異なるハンドラにブレークポイントを設定 
    set_breakpoint({ className: "com.example.Api1Controller", line: 30 })
set_breakpoint({ className: "com.example.Api2Controller", line: 25 })
  2. 両方のエンドポイントにリクエスト（例: curl で）

    両スレッドがそれぞれのブレークポイントで独立して停止します。

  3. 各スレッドを確認 
    get_threads({})  // 全スレッドとステータスを表示
get_stack_trace({ threadId: "42" })
get_variables({ threadId: "42" })
get_stack_trace({ threadId: "43" })
get_variables({ threadId: "43" })
  4. 各スレッドを独立してステップ実行 
    step_over({ threadId: "42" })  // スレッド 43 は停止したまま
step_over({ threadId: "43" })  // スレッド 42 は停止したまま

変数の確認

ローカル変数、オブジェクトのフィールド、配列の要素を確認できます：

// 現在のフレームのローカル変数を取得
get_variables({ threadId: "42", frameIndex: 0 })
// 出力例:
//   String name = "Alice"
//   int count = 5
//   UserService userService = Object@1234

// オブジェクトの詳細を確認
inspect_object({ objectId: "1234" })
// 出力例:
//   String name = "UserService"
//   Map<String, User> cache = Object@5678

// 配列の要素を確認
inspect_array({ arrayId: "9012", startIndex: 0, length: 10 })

プロトコルスタック

本サーバーは JDWP (Java Debug Wire Protocol) を TCP 上で直接実装しており、JDI (Java Debug Interface) に依存しません。これにより、純粋な Node.js 環境で動作します。

レイヤー技術役割
クライアントインターフェースMCP (stdio)AI アシスタントとの通信
デバッグプロトコルJDWP (TCP)JVM デバッグ用バイナリプロトコル
対象JVM (JVMTI エージェント)デバッグ対象の Java/Kotlin アプリケーション

トラブルシューティング

接続が拒否される

対象 JVM が JDWP エージェント付きで起動されていて、ポートが一致していることを確認してください：

-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005

ブレークポイント設定時にクラスが見つからない

  • 完全修飾クラス名を使用してください（例: MyClass ではなく com.example.MyClass）
  • クラスが JVM にロードされている必要があります。遅延ブレークポイントを使用する場合は、先にクラスのロードをトリガーしてください。

ポートが既に使用中

他のプロセスがポート 5005 を使用している可能性があります。停止するか、別のポートを指定してください：

launch({ projectDir: "...", port: 5006 })

launch は成功するが connect が失敗する

launch が「JDWP ポートがリッスンしていない」と報告した場合、プロセスは起動したものの JDWP デバッグポートが公開されていません。主な原因：

  • ビルドシステムのタスクが JDWP 引数を対象 JVM に渡していない（例: exec-maven-plugin のないプレーン Maven プロジェクト）
  • アプリケーションの起動に 10 秒以上かかる場合。この場合は待ってから手動で connect を試してください。

Gradle デーモンの競合

本サーバーは Gradle init script を使用して JavaExec タスクにのみデバッグフラグを注入します。Gradle デーモン自体は影響を受けません。問題が発生した場合は、起動前に ./gradlew --stop を実行してください。
